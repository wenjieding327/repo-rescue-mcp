# RepoRescue 完整产品需求

## 1. 产品目标

让用户无需懂命令、环境配置或提示词工程，只需粘贴代码、报错、文件或仓库，就能得到“理解 → 诊断 → 修复 → 实际验证 → 可复用产物”的闭环。

主宣传语：

> 无论是一段代码、一个报错，还是整个项目，RepoRescue 都能帮你看懂、修复并运行验证。

它不是“什么代码都能写”的万能聊天助手，核心差异是生成式修复与执行证据绑定。

## 2. 三层入口

### A. 普通代码急救：负责获客与热度

允许直接粘贴：

- 一段代码
- 报错文本或截图
- 作业题
- 某个函数
- 从零编程需求

输出：

- 小白能懂的根因
- 完整可复制代码或最小 Diff
- 修改前后对比
- 安全沙箱中的实际运行结果
- 没有预期输出时明确标记为“候选代码已运行”

### B. 文件级救援：负责留存

支持 `.py`、`.ipynb` 等，分析：

- 语法和逻辑问题
- 输入输出错误
- 依赖缺失
- 测试结果
- 文件结构与执行顺序
- 从哪里开始阅读

### C. 项目复现：负责技术护城河

输入公开 GitHub 项目，完成：

- 仓库与 commit 固定
- README、目录和入口分析
- 依赖与环境构建
- 实际执行和测试
- 失败归因
- 自动修复与再次验证
- 复现报告、环境文件和代码补丁

## 3. 标准闭环

输入识别 → 代码/仓库分析 → 环境构建 → 修改前执行 → 失败定位 → 最小修复 → 修改后执行 → 证据分级 → 生成报告。

修复失败时可在预算内继续迭代，但必须显示最终真实状态。

## 4. 验证等级

- P0：未执行，仅生成建议
- P1：代码/仓库可读取
- P2：依赖可解析或安装
- P3：指定测试范围通过
- P4：官方示例或 Demo 复现
- P5：论文核心指标在声明条件下复现

禁止用统一的 `verified: true` 混淆这些等级。

## 5. 真实性与安全

- 只有原代码在同一用例失败、候选代码通过，才能说“已验证修复”。
- 不得虚构 commit、日志、退出码、测试数或论文指标。
- 不得删除测试、弱化断言或硬编码答案来制造通过。
- 区分 AI 推测和工具执行事实。
- 默认拒绝危险系统调用、无限循环、未授权网络访问和敏感信息读取。
- 不能自动读取用户 Windows 电脑；需要本机信息时只给明确的只读探测命令。

## 6. 新手优先结果页

首屏只显示：

- 成功 / 失败 / 未执行
- 复现等级
- 一句话问题
- 改了什么
- 修改前怎样失败 → 修改后通过几个用例
- 完整可复制代码
- 预计节省时间

哈希、backend、完整日志和底层参数默认折叠。

## 7. 生成式体验

- 用户不需要自己提取 Python 版本、依赖或报错字段。
- Agent 主动判断任务类型、补充合理测试、生成修复代码。
- 用户描述不完整时先完成可安全推断的部分，再问一个真正阻塞的问题。
- 从零写代码与修复代码必须区分验证措辞。
- 保留用户原接口，优先最小改动。

## 8. 项目分析能力

- 识别语言、框架、入口、测试命令和关键目录。
- 从 README、配置文件和 CI 中推断官方运行路径。
- 锁定 commit 和依赖版本。
- 区分代码错误、环境错误、数据缺失、文档缺失、外部服务缺失。
- 检测 Python/CUDA/PyTorch/操作系统兼容性。

## 9. 自动修复能力

- 依赖版本冲突
- 导入与 API 迁移
- 路径、配置和随机种子
- 输入输出与标签映射
- README 缺失步骤
- 测试失败的最小补丁

所有修复输出 Diff；高风险或改变算法语义的修改需用户确认。

## 10. 可交付产物

- 可复现报告
- `requirements.lock`、环境 YAML 或容器配置
- 修改前后 Diff / patch
- 执行日志与结构化证据
- 测试范围与验证等级
- 可选 GitHub Pull Request

## 11. 三个比赛演示案例

1. 初学者单段代码：10 秒内定位索引错误、修复并验证。
2. 中型项目：依赖冲突导致失败，自动建立兼容环境并通过官方测试。
3. 科研仓库：README 不完整且数据路径错误，完成诊断、修复、官方 Demo，并清楚标记论文指标是否达到。

演示必须有真实的“失败 → 诊断 → 修复 → 成功”，不能只展示聊天。

## 12. 热度和传播设计

- 首页无需注册即可粘贴一个短片段体验。
- 提供“报错截图”“作业救急”“GitHub 跑不通”三个明显入口。
- 生成可分享的救援卡片：问题、修复、验证、节省时间。
- 展示真实前后对比，而不是堆工具术语。
- 允许收藏和复用环境/修复模板。
- 用短视频演示“原来两天跑不通，几分钟定位并修复”。

## 13. 量化指标

- 首次有效结果时间
- 片段修复验证成功率
- 问题定位准确率
- 自动修复成功率
- P3/P4/P5 各级成功率
- 平均复现时间和相对人工节省时间
- 次日/七日留存
- 分享率和二次使用率

## 14. 能力边界

必须公开：

- 支持语言与文件类型
- 仓库大小与运行时长限制
- 是否支持 GPU、Docker、数据库、网络和私有仓库
- 数据集与模型权重获取方式
- 沙箱安全限制
- 哪些结论只是推测，哪些经过执行验证

## 15. 架构与可迁移性

- 生成、路由和解释留在 Agent 层。
- 执行、验证、证据与仓库检查做成独立 MCP/API。
- 提示词、测试集和报告模板全部版本控制。
- 讯飞只做编排外壳，避免把核心能力锁死在专属节点。
- 同一 MCP 可接讯飞、OpenClaw、Codex Skill 或网页前端。

## 16. 开发优先级

### 初赛/当前

- 片段生成、修复、真实验证
- 公开仓库检查和白名单复现
- 新手输出与真实性分级
- 至少两个可重复演示

### 复赛

- 文件/Notebook 上传
- 隔离环境和更复杂依赖
- 自动补丁循环
- 官方 Demo 复现
- 报告、环境文件和分享卡片

### 长期

- GPU/Docker/私有仓库
- 论文指标对齐
- PR 与团队协作
- 复现模板市场和实验室批量评估
